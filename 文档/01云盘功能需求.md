# 网络云盘系统功能实现细节文档

## 项目概述

### 技术架构

> **注意**: 详细的技术栈信息和兼容性分析已迁移到专门的文档：[网络云盘系统-技术栈与兼容性分析.md](./网络云盘系统-技术栈与兼容性分析.md)

本系统采用现代化的前后端分离架构，主要技术栈包括：
- **前端**: React 18 + TypeScript + Ant Design + Redux Toolkit
- **后端**: Go 1.24.6 + Gin 1.10.0 + GORM 1.30.1 + WebSocket
- **数据库**: MySQL 8.0 + MongoDB 6.0 + Redis 7.0+
- **搜索引擎**: MySQL 8.0+ 全文索引
- **消息队列**: Redis Pub/Sub + Stream 7.0+
- **部署**: Docker + Nginx

## 功能实现细节

## 1. 用户认证系统

### 1.1 邮箱注册流程
**前端交互与验证：**
- 表单验证机制：
  * 邮箱验证：实时检测格式（正则匹配`^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$`），失去焦点时提示格式错误
  * 密码强度检测：实时评估（弱/中/强），至少8位且包含大小写字母（A-Z/a-z）、数字（0-9）
  * 支持显示强度进度条及具体要求提示（如"请添加大写字母"）
  * 重复密码校验：实时对比两次输入，不一致时高亮提示
  * 防重复提交：点击"获取验证码"后按钮置灰60秒（倒计时显示），避免频繁请求

**后端处理流程：**
- 前置校验：
  * 接收注册请求后，先检查邮箱是否已存在（MySQL users表email字段唯一索引校验）
  * 防恶意注册：限制同一IP 24小时内最多3次注册尝试，超限则临时封禁
  * Redis记录`register_limit:{ip}`（IP地址格式：IPv4使用点分十进制如`192.168.1.1`，IPv6转小写如`2001:db8::1`），过期24小时
- 验证码生成与发送：
  * 生成6位随机数字验证码（000000-999999）
  * 通过Redis存储（key: `email_verify:{email}`, value: 验证码+创建时间戳，过期5分钟）
  * 邮件模板：采用HTML格式，包含验证码（加粗显示）、有效期提示、平台logo及客服联系方式
  * 发送重试：若SMTP服务临时失败，自动重试2次（间隔3秒），仍失败则返回提示
- 注册完成与初始化：
  * 验证码验证通过后，使用bcrypt算法加密密码（cost=12，平衡加密强度与性能）
  * 存储至users表，初始化用户信息：默认角色user、默认存储空间10GB、创建时间戳、最后登录时间
  * 返回注册成功响应，附带"前往登录"引导链接

### 1.2 找回密码机制
**验证流程设计：**
- 邮箱验证阶段：
  * 用户输入邮箱后，前端提示"若邮箱已注册，将发送验证码"（避免泄露账号是否存在）
  * 后端校验邮箱存在性，存在则生成6位验证码，存储至Redis（key: `password_reset:{email}`，过期10分钟）
  * 邮件内容：明确标注"密码重置验证码"，提醒"仅用于重置密码，勿向他人提供"
- 密码重置阶段：
  * 前端验证验证码格式（6位数字），后端校验验证码有效性（匹配Redis记录且未过期）
  * 验证通过后，跳转至新密码设置页，新密码需满足注册时的强度规则（前后端双重校验）
  * 密码更新：使用bcrypt重新加密新密码，更新users表中对应记录，并删除Redis中的验证码
  * 安全增强：更新密码后，自动失效该用户所有旧的refresh token，强制旧设备重新登录

### 1.3 登录认证
**登录验证机制：**
- 登录方式支持：
  * 兼容"邮箱+密码"或"用户名+密码"登录，后端统一通过users表email或username字段查询用户
  * 密码验证：使用bcrypt.CompareHashAndPassword比对输入密码与存储的加密哈希
- 防暴力破解：
  * 失败次数限制：同一账号连续5次密码错误，临时锁定15分钟
  * Redis记录`login_lock:{user_id}`（user_id格式：int64数字直接拼接如`login_lock:12345`），过期15分钟
  * 锁定提示：前端显示"账号已临时锁定，请15分钟后重试"，避免泄露具体信息

**Token管理策略：**
- Token生成规则：
  * Access Token：基于JWT生成，有效期2小时，payload包含user_id、username、role、exp
  * 签名密钥使用服务器环境变量存储
  * Refresh Token：生成32位随机字符串（UUID），有效期7天
  * 存储至Redis（key: `refresh_token:{user_id}`，value: token哈希值）
- Token验证与刷新：
  * 请求验证：前端在Authorization头携带Bearer {access_token}，后端中间件解析并校验
  * 自动刷新：Access Token过期时，前端携带Refresh Token请求新的Access Token
  * Refresh Token采用滑动过期，每次刷新后有效期重置为7天
  * 失效处理：Refresh Token过期/被手动登出时，删除Redis记录，前端跳转至登录页

**安全与体验增强：**
- 安全防护：
  * 全站HTTPS传输，防止Token与密码被窃听
  * 登录请求携带CSRF Token，防止跨站请求伪造
  * JWT不包含敏感信息，降低泄露风险
- 用户体验优化：
  * "记住我"选项：勾选后Refresh Token有效期延长至30天
  * 登录状态保持：
  - Access Token存储在内存（避免XSS攻击，页面刷新时通过Refresh Token重新获取）
  - Refresh Token存储在httpOnly Cookie中（设置secure、SameSite=Strict属性，防止XSS和CSRF攻击）
  - localStorage仅存储非敏感的用户偏好设置（如主题、语言等）
  * 异常提示：区分不同错误场景，提供针对性引导（如"忘记密码？点击找回"）

## 2. 文件管理系统

### 2.1 文件上传功能
**分片上传实现：**
- 前端使用File API将大文件切分为动态大小的分片：
  * 小文件（<10MB）：可不分片或使用较大分片（10MB）
  * 大文件（>1GB）：使用较小分片（2MB）
  * 根据网络状况（通过navigator.connection API）动态调整分片大小
- 每个分片包含：文件MD5、分片索引、总分片数、分片MD5、分片大小
- 上传前先调用预上传接口，后端检查：
  * 文件是否已存在（秒传功能）
  * 返回已上传的分片列表（部分上传场景）
  * 返回建议的分片大小和临时上传令牌（有效期24小时）
- 使用Web Worker计算MD5，避免阻塞主线程
- 优化文件MD5计算：通过所有分片MD5串联后再哈希生成
- 并发上传分片（最大并发数6），每个分片上传到临时目录
- 后端为每个上传任务生成唯一的upload_id（格式：文件MD5前8位_时间戳_随机数，如：`a1b2c3d4_1703123456_xyz789`）
- 分片上传安全校验：
  * 所有请求需携带临时令牌验证
  * 后端校验分片MD5和分片大小，防止数据篡改
  * 文件类型安全校验（按《后端安全设计》要求）：
    - 前端上传时附加文件MIME类型和文件名
    - 后端使用`file --mime-type`命令二次校验真实文件类型
    - 检查文件头魔数（Magic Number）防止伪装文件（如.exe改名为.jpg）
    - 维护文件类型白名单，拒绝可执行文件和脚本文件
    - 对图片文件进行额外校验：检查EXIF信息，防止恶意代码注入

**测试覆盖要求（关联测试规范）：**
- 前端测试要求（按《前端测试规范》）：
  * 分片上传的断点续传功能需E2E测试覆盖关键路径
  * 单元测试覆盖文件MD5计算、分片合并逻辑，覆盖率≥80%
  * 组件测试覆盖上传进度展示、错误处理UI组件，覆盖率≥90%
- 后端测试要求（按《后端测试规范》）：
  * 合并分片的原子性通过集成测试验证，覆盖率≥95%
  * 单元测试覆盖文件类型校验、安全检查逻辑，核心业务逻辑覆盖率≥90%
  * API接口测试覆盖所有上传相关接口，覆盖率≥95%

**断点续传实现：**
- Redis使用Set类型存储上传进度（key: `upload_progress:{upload_id}`），判断分片是否已上传的时间复杂度为O(1)
- 前端断点续传增强：
  * localStorage缓存upload_id和已上传分片信息（结合文件MD5作为key）
  * 页面刷新后可恢复上传状态，无需重新计算MD5
  * 显示整体上传进度和预估剩余时间
- 分片上传错误处理：
  * 失败分片自动重试（最多3次，指数退避间隔：1s→2s→4s）
  * 重试前校验分片是否已上传成功（避免无效重试）
  * 多次失败后提供"暂停"和"重新开始"选项
- 所有分片上传完成后，前端主动调用合并接口
- 后端使用分布式锁确保合并原子性，合并完成后：
  * 计算文件MD5，验证文件完整性
  * 文件上传到阿里云OSS，删除临时分片文件
- 临时文件管理：
  * 合并成功后异步删除临时分片
  * 定时任务清理过期未完成的临时分片（超过24小时）
  * 前端取消上传时调用取消接口，立即清理临时文件

**异步任务处理优化：**
- 任务队列可靠性增强：
  * 重要任务采用"先存MySQL，再入Redis队列"的双写模式
  * Redis队列开启AOF持久化，降低数据丢失风险
  * 按任务类型拆分队列（task_queue:high/task_queue:low）
  * 消费端限流：通过Redis Semaphore控制同类型任务的最大并发数
- 任务生命周期管理：
  * 任务状态细分：pending、processing、success、failed、retrying
  * 失败任务采用指数退避重试（30s→2min→10min）
  * 超过最大重试次数的任务进入死信队列（task_queue:dead）
- 任务监控与可观测性：
  * 定期检查各任务队列长度，超过阈值触发告警
  * MySQL tasks表增加error_msg（失败原因）、process_time（处理耗时）字段
  * 为每个任务关联upload_id，实现全链路追踪

**任务类型与优先级：**
- 高优先级任务：
  * 文件病毒扫描（安全性要求高）
  * 存储空间统计更新（影响用户体验）
  * 邮件发送任务（时效性要求高）
- 低优先级任务：
  * 图片缩略图生成
  * 视频封面图生成
  * 文档预览图生成

**OSS交互优化：**
- 分片直传OSS：
  * 大文件（>100MB）使用OSS分片上传API
  * 前端直接向OSS上传分片（通过应用服务器签名URL授权）
  * 应用服务器仅记录分片状态和负责最终合并，减少服务器磁盘占用
- OSS上传校验：
  * 文件合并后上传OSS时，同步计算文件MD5并与OSS的ETag对比
  * 在OSS中设置MD5元数据，确保上传文件完整无误

**数据一致性保证衔接：**
- OSS上传完成后的数据同步流程：
  * 步骤1：更新MySQL的files表（oss_key、file_size、upload_status='completed'）
  * 步骤2：同步更新MongoDB的file_analysis集合（file_id、文件元数据）
  * 步骤3：通过Redis Stream发送一致性检查任务到队列`data_consistency:check`
  * 步骤4：消费端验证MySQL与MongoDB数据一致性，异常时触发数据修复机制
- 分布式事务处理：
  * 使用DTM Saga模式确保OSS上传→MySQL更新→MongoDB同步的事务一致性
  * 任一步骤失败时，自动执行补偿操作（如删除OSS文件、回滚数据库记录）
- 临时文件清理策略：
  * 合并完成后异步删除临时分片
  * 定时任务每小时清理超过24小时未合并的临时分片
  * 用户超过1小时未操作时，前端主动调用取消上传接口

**异步任务实现规范：**
- Redis Stream消息队列处理：
  * 清理任务发送到低优先级队列`task_queue:low`，避免影响核心业务
  * 消息格式：`{task_type: "cleanup_temp_files", upload_id: "xxx", file_paths: ["path1", "path2"]}`
  * 消费端按《后端项目结构》的任务处理规范执行：最大并发度5，失败重试3次

**维护指南联动机制：**
- 任务监控与告警：
  * 清理任务失败时，触发《后端维护指南》的故障告警机制
  * 队列长度超过1000时，联动《技术选型》的监控关键性能指标
  * 通过Prometheus metrics记录清理成功率和平均处理时间
- 故障诊断支持：
  * 临时文件清理失败时，自动记录到《后端维护指南》的日志分析系统
  * 上传任务异常时，生成详细的诊断报告供运维人员分析
  * 性能指标异常时，自动触发《系统架构设计》的降级策略

**用户体验优化：**
- 上传进度精细化：
  * 展示"已完成x/y个分片"和"已上传字节数/总字节数"
  * 结合已上传分片的平均速度，动态预估剩余时间
- 任务状态反馈：
  * 文件上传完成后，通过WebSocket接收异步任务处理状态
  * 展示进度条或状态图标（如"病毒扫描中→扫描完成→缩略图生成中→全部完成"）
- 大文件预处理提示：
  * 超大文件（>1GB）选择后先提示"正在计算MD5（预计x秒）"
  * 通过Web Worker监听计算进度并显示

**文件夹上传实现：**
- 前端技术实现：
  * 使用HTML5 `<input type="file" webkitdirectory>` 属性支持选择文件夹
  * 兼容性处理：对不支持的浏览器提供"压缩文件夹后上传"的引导
  * 大文件夹读取优化：
    - 使用requestIdleCallback或分批异步读取（每批100个文件）
    - 读取过程中显示"正在解析文件夹结构（x/y个文件）"的进度提示
    - 限制单文件夹最大文件数（1000个），超过时提示拆分
  * 构建文件树结构，包含路径信息和父子关系
- 后端处理流程：
  * 路径注入防护：规范化处理所有路径，防止../等字符遍历到根目录外
  * 目录创建的依赖管理：
    - 前端按"广度优先"顺序提供目录列表（先父后子）
    - 后端通过队列按层级顺序创建目录或使用数据库事务确保原子性
  * 目录重名冲突处理：
    - 自动重命名（如"文件夹1→文件夹1(1)"）
    - 合并目录（将新文件添加到现有目录）
    - 提示用户选择处理方式（适合重要目录）
  * 为每个文件生成唯一的upload_id，关联到对应目录
- 断点续传支持：
  * 使用IndexedDB存储文件夹状态（替代localStorage），支持更大容量
  * 状态一致性校验：恢复上传时同时校验后端"已上传文件列表"
  * 支持从任意位置恢复文件夹上传
  * 已完成的文件和文件夹不再重复上传
- 上传进度展示：
  * 整体进度：已上传文件数/总文件数和已上传字节数/总字节数
  * 当前文件进度：分片上传百分比
  * 文件夹层级结构可视化展示，支持展开/折叠
  * 对已完成/部分完成/未开始的文件夹用不同颜色标识

**MongoDB 6.0 LTS 版本适配说明：**
- 文件夹层级查询基于MongoDB 6.0的聚合管道实现，避免使用8.0+新增语法
- 文件关系存储使用MongoDB 6.0原生支持的多文档事务，确保文件夹操作的原子性
- 索引策略采用6.0兼容的复合索引（如`{parent_id: 1, name: 1}`），避免8.0+的新索引特性
- 查询优化基于6.0的查询优化器和执行计划，不依赖8.0+的性能提升特性

**多文件上传实现：**
- 批量文件选择：
  * 支持Ctrl/Shift多选或拖拽多个文件
  * 预处理阶段显示文件列表，支持编辑和排序
- 上传前过滤与校验：
  * 检测文件大小是否超过限制（如单个文件>10GB时提示）
  * 过滤不支持的文件类型（根据业务需求配置）
  * 批量校验文件名（自动替换特殊字符并提示）
  * 重复文件检测：通过"文件名+大小"或"文件MD5预计算"判断
- 上传队列管理：
  * 动态并发调整：
    - 根据网络状况（navigator.connection.downlink）自适应调整并发数
    - 根据服务器负载动态调整（通过接口返回max_concurrent）
  * 队列优先级精细化：
    - 用户手动置顶：允许拖拽调整队列顺序
    - 类型优先级：如"文档"优先于"视频"
    - 失败重试插队：失败后重试的文件优先级更高
  * 支持暂停、恢复、取消单个或全部上传任务
- 智能上传策略：
  * 小文件（<10MB）优先上传，提高完成感
  * 相似文件（如同类型文件）连续上传，优化服务器处理
  * 失败文件自动重试，最多3次
- 批量上传状态管理：
  * 整体进度条：总体完成百分比
  * 文件列表：每个文件的状态（等待中、上传中、已完成、失败）
  * 批量操作：支持"全选/反选"文件，批量暂停、恢复、取消
  * 上传完成后的批量处理选项（如全部分享、移动到指定文件夹）
  * 失败文件汇总展示（如"3个文件上传失败，点击重试"）
- 后台上传支持：
  * 利用Service Worker实现页面关闭后继续上传（HTTPS环境）
  * 重新打开页面时自动恢复状态
  * 大文件内存优化：上传完成后及时释放File对象，清理临时URL

**多文件夹批量上传：**
- 高级选择功能：
  * 支持同时选择多个文件夹
  * 拖拽整个文件夹组到上传区域
- 文件夹结构的高效处理：
  * 重复文件夹的智能合并：
    - 合并模式：合并所有子文件，同名文件按规则处理
    - 独立模式：自动重命名为不同名称，保持结构独立
    - 跳过模式：只上传第一个，跳过后续同名文件夹
  * 深层目录的可视化优化：
    - 可折叠树形结构，支持展开/折叠子文件
    - hover时显示完整路径
    - 使用不同颜色标识完成状态
  * 维持原始文件夹层级关系
- 上传调度的优先级控制：
  * 文件夹级别的优先级配置：
    - 允许用户通过拖拽或"优先级滑块"设置文件夹上传顺序
    - 支持标记特定文件夹为高/中/低优先级
  * 资源占用平衡：
    - "文件夹轮询"策略：每个文件夹分配并发线程
    - 确保资源分配公平，避免大文件占用所有线程
  * 文件按大小和类型优化上传顺序
- 断点续传的粒度控制：
  * 多级恢复选项：
    - 整个批次恢复：恢复所有未完成文件夹
    - 单个文件夹恢复：仅恢复选定文件夹

  * 状态备份优化：
    - 每完成一个文件后备份该文件状态
    - 每5分钟全量备份一次批次状态

- 后端存储优化：
  * 对象存储直接按"文件夹路径+文件名"作为对象键
  * 无需在服务器创建实际目录，节省磁盘空间

### 2.2 文件夹管理
**创建文件夹：**
- 权限与校验增强：
  * 权限粒度细化：区分权限类型（可创建/仅查看/管理员）
  * 普通用户只能在自己的目录下创建，管理员可在任意目录创建
  * 名称校验规则：
    - 限制特殊字符（如/\:*?"<>|），避免与文件系统或URL冲突
    - 限制名称长度（最多255字符），防止数据库字段溢出
    - 禁止使用系统保留名（如..、.，避免路径解析歧义）
- 多级文件夹批量创建优化：
  * 支持通过"路径字符串"一次性创建多级文件夹（如/a/b/c）
  * 后端解析路径时，按层级检查父目录是否存在
  * 若中间某级不存在，自动创建（类似mkdir -p）
  * 若某级已存在且不是文件夹（是文件），则返回冲突错误
- 在files表中插入记录（type='folder', size=0, parent_id=父ID）

**重命名操作：**
- 数据结构优化：
  * 不存储完整路径，改用"父ID+名称"动态计算路径
  * 数据库files表仅存parent_id和name，不存path字段
  * 路径通过递归查询父ID链动态生成
  * 重命名时，仅需更新当前文件夹的name字段，无需更新子文件
- 冲突处理与用户体验：
  * 前端输入新名称时，通过防抖请求实时校验同级目录是否存在同名文件/文件夹
  * 若文件夹包含大量子文件（1000+），重命名前提示"该操作可能需要几秒"
  * 提供"重命名并移动"选项，允许同时修改名称和父目录

**删除操作：**
- 软删除实现优化：
  * 设置is_deleted=true，文件移入回收站
  * 异步递归标记：删除请求仅标记当前文件夹，子文件通过异步任务处理
  * 查询时过滤：通过parent_id和is_deleted的联合索引，自动过滤"父文件夹已删除"的子文件
- 回收站与清理机制：
  * 数据库files表增加deleted_at（删除时间）和original_parent_id（删除前的父ID）
  * 回收站文件保留期限可配置（默认30天），到期后自动触发"彻底删除"任务
  * 大容量删除提示：删除大量文件时，显示进度和预计完成时间
- 恢复机制：
  * 恢复时，若原路径的父文件夹已删除，提示"原路径不存在，是否恢复到根目录/指定目录"
  * 恢复时，若目标路径存在同名文件，提供"覆盖/重命名/取消"选项
- 暂不删除OSS中的文件，等待彻底删除时再清理

**文件夹移动操作：**
- 移动验证：
  * 验证目标父文件夹是否存在且未删除
  * 检查目标路径是否存在同名文件夹（处理冲突）
  * 验证用户是否有原文件夹的"移动权限"和目标文件夹的"写入权限"
- 高效实现：
  * 移动时仅更新文件夹的parent_id，无需修改子文件信息
  * 支持批量移动多个文件/文件夹
  * 移动过程中显示进度，支持取消操作

**权限继承机制：**
- 文件夹创建时默认继承父文件夹的权限（如"谁可以查看/编辑"）
- 支持"打破继承"：为文件夹单独设置权限，子文件默认继承该文件夹权限
- 权限变更时提供选项："仅应用于此文件夹"或"应用于此文件夹及所有子项"
- 权限冲突解决：当文件同时存在于多个共享文件夹时，取权限范围最大值

### 2.3 文件下载功能
**单文件下载的断点续传优化：**
- 协议层面支持：
  * 基于HTTP Range请求头实现断点续传
  * 前端记录已下载字节范围，下次请求时通过`Range: bytes=102401-`获取剩余部分
  * 后端返回206 Partial Content，携带Content-Range和ETag
- 后端校验机制：
  * 验证Range请求范围的合法性（不超过文件总大小）
  * 通过ETag判断文件是否被修改（若修改则需重新下载）
  * 对每个分片请求返回独立的校验值（如CRC32）
- 大文件分片下载策略：
  * 前端将大文件（>1GB）主动拆分为多个固定大小分片（如100MB/片）
  * 并发请求不同分片（控制并发数，如3-5个）
  * 下载完成后在本地合并，并校验每个分片的完整性

**批量下载的打包优化：**
- 异步打包避免阻塞：
  * 前端发起批量下载请求后，后端返回"任务ID"并将打包任务放入队列
  * 打包完成后，通过WebSocket或轮询通知前端"下载已准备好"
  * 打包过程中，前端显示实时进度（已处理x/y个文件）
- 流式打包减少内存占用：
  * 采用流式写入（如Go的archive/zip包），逐个添加文件到ZIP流
  * 避免将所有文件加载到内存，防止OOM
- 超大批量的拆分策略：
  * 批量下载包含1000+文件时，自动拆分为多个ZIP包（如每200个文件一个）
  * 前端显示"共x个包，正在下载第1个"

**临时下载链接的安全增强：**
- token生成规则：
  * 包含核心信息：用户ID、文件ID、过期时间（timestamp）
  * 防篡改签名：用服务器密钥进行HMAC-SHA256加密
  * 示例：`https://domain.com/download?token=user123_file456_1690000000_sign=xxx`
- 权限二次校验：
  * 下载请求到达后端时再次校验用户权限
  * 检查token是否已被使用（防止重复下载，适合敏感文件）
- 链接时效精细化：
  * 小文件（<100MB）：10分钟有效期
  * 大文件（100MB-1GB）：1小时有效期
  * 超大文件（>1GB）：24小时有效期

**防滥用与限流：**
- 单IP/用户限流：
  * 通过Redis记录单位时间内的下载次数/流量
  * 超过限制则返回429 Too Many Requests
- 下载行为异常检测：
  * 监控短时间内重复下载相同文件
  * 检测异常下载速度，触发二次验证（如验证码）

**CDN加速的深度适配：**
- 静态文件缓存策略：
  * 公开文件：CDN缓存有效期较长（如7天），`Cache-Control: public, max-age=604800`
  * 私有文件：CDN仅作为传输加速层，不缓存内容，`Cache-Control: private, no-store`
- 动态内容的CDN协同：
  * 对"相同文件组合"的ZIP包进行短期缓存（如1小时）
- 多CDN智能切换：
  * 配置多个CDN厂商，监控各节点响应速度
  * 自动将用户请求路由到最优节点

**下载速度与并发控制：**
- 基于用户等级的动态限速：
  * 令牌桶算法实现限速（每秒生成N个令牌，每个令牌对应1KB流量）
  * 不同等级用户配置不同令牌生成速率（普通用户1MB/s，VIP用户5MB/s）
- 并发连接控制：
  * 全局限制：服务器配置最大并发下载数（如500）
  * 单用户限制：同一用户最多同时下载3个文件

**下载状态的精细化反馈：**
- 实时进度展示：
  * 前端通过XMLHttpRequest的progress事件获取已下载字节数
  * 显示实时下载速度和预估剩余时间
- 批量下载的明细反馈：
  * 展示每个文件的状态（"等待打包"、"已加入包中"、"打包失败"）
  * 失败文件单独标注原因

**容错与恢复机制：**
- 下载中断的自动重试：
  * 网络临时中断时，等待3秒后自动重试（最多3次）
  * 重试失败后，显示"下载中断，点击恢复"按钮
- 文件损坏的校验与补救：
  * 下载完成后，计算文件校验值与后端提供的校验值比对
  * 不一致时提示"文件可能损坏，是否重新下载"

**下载队列的灵活管理：**
- 任务优先级调整：
  * 用户可拖拽调整任务顺序或标记"优先下载"
- 跨会话恢复：
  * 用IndexedDB存储下载任务状态
  * 重新打开页面时自动恢复队列

**统计与日志的增强：**
- 核心日志字段：
  * 下载来源（直接下载/分享链接/API调用）
  * 终端信息（设备类型、浏览器）
  * 下载结果细分（成功/网络中断/权限不足/文件不存在）
  * CDN相关（命中节点、传输耗时、是否缓存命中）
- 数据分析应用：
  * 热门文件TOP10，优化CDN缓存策略
  * 用户下载高峰时段，提前扩容服务器资源
  * 异常下载行为预警

### 2.5 文件同步功能
**多设备同步：精度与冲突处理**
- 增量同步判断升级：

  * 仅同步修改的块（类似Rsync算法），大幅减少大文件同步流量
  * 同步依据补充：除MD5和修改时间，增加"文件大小"校验
  * 避免因修改时间异常（如跨时区设备）导致的误判
- 冲突处理精细化：
  * 冲突检测场景扩展：包括"同一文件多设备修改"、"文件被删除同时被修改"、"文件夹重命名同时子文件被修改"等
  * 冲突解决选项：
    - 按设备优先级（如"PC修改优先于手机"）
    - 按修改时间（如"保留最新修改"）
    - 手动合并（适合文本/文档，提供在线对比合并工具）
  * 冲突提示增强：显示冲突文件的"修改摘要"，帮助用户决策
- 支持Windows、macOS、Linux客户端和移动端APP（iOS/Android）

**同步策略：适应性与资源控制**
- 实时同步实现优化：
  * 不同系统的文件监听适配：
    - Windows用ReadDirectoryChangesW
    - macOS用FSEvents
    - Linux用inotify
  * 防抖处理：短时间内多次修改合并为一次同步请求（延迟1秒）
- 定时同步：按设定间隔同步（可配置，如每小时）
- 手动同步：用户主动触发同步
- 选择性同步体验提升：
  * 可视化选择界面：以树形结构展示文件夹
  * 勾选时显示"预计同步大小"和"包含文件数"
  * 智能推荐：根据访问频率推荐"不同步"的文件夹
  * 部分同步支持：
    - 仅同步最近修改的N个文件
    - 仅同步特定类型文件（如.docx）
- 带宽与资源控制：
  * 同步限速策略：支持"工作时间限速+空闲时间全速"
  * 电量适配：移动端低电量时自动暂停同步

**同步可靠性保障**
- 断点续传与重试：
  * 同步任务中断后，记录已同步的块/文件，恢复连接后从断点继续
  * 同步失败重试策略：
    - 临时错误重试3次（间隔1s→3s→5s）
    - 持久错误暂停并提示用户处理
- 数据一致性校验：
  * 定期全量校验（每周）：对比本地文件与云端的"块级MD5"
  * 修复因意外导致的不一致（如本地文件损坏）
  * 同步日志详细记录：包含同步时间、文件路径、操作类型、状态、失败原因
  * 支持用户查看和问题排查

  * 原始路径存在性校验：检查original_parent_id对应的文件夹是否存在且未被删除
  * 若原文件夹在回收站中，提示"是否先恢复该文件夹再恢复文件？"
  * 对深层路径，自动递归恢复所有必要的祖先文件夹，确保文件回到原始位置
- 同名文件冲突处理：
  * 提供用户选择项："覆盖同名文件"、"保留两者（自动重命名）"、"取消恢复"
  * 批量恢复时，对所有同名文件统一应用用户选择的策略
- 批量恢复与大文件夹恢复优化：
  * 异步恢复机制：通过Redis队列处理递归恢复，避免HTTP超时
  * 前端通过轮询或WebSocket获取恢复进度（"已恢复30/120个文件"）
  * 恢复失败的容错处理：记录失败原因，提供"重试恢复失败文件"按钮
  * 恢复中断后，已恢复的文件保持有效，未恢复的文件可继续处理

**彻底删除的原子性保障：**
- 事务与补偿机制：
  * 采用"数据库记录删除→OSS文件删除→存储空间更新"的事务流程
  * 任一环节失败则回滚，删除失败的文件标记为"删除异常"
  * "删除异常"文件在回收站中特殊标识（红色警告图标）
- 批量彻底删除的资源控制：
  * 单次批量删除限制文件数量（最多500个），超过则提示"请分批删除"
  * 大文件（>1GB）的彻底删除放入异步队列后台处理
  * 更新用户已使用存储空间统计
  * 删除相关的分享记录和权限记录

**自动清理机制的精细化：**
- 清理任务的效率提升：
  * 预计算待清理文件：低峰期查询过期文件ID存入Redis Set，分批删除
  * 优先删除"已彻底删除过但未清理成功"的文件，减少无效数据堆积
- 存储空间不足时的智能清理：
  * 用户存储空间使用率>90%时触发"紧急清理"
  * 按"删除时间最早→文件体积最大"的优先级清理，直到使用率<80%
  * 清理前10分钟发送实时通知，提供"取消自动清理"选项
- 回收站文件默认保留30天（管理员可配置）

**回收站权限与操作控制：**
- 多层权限校验：
  * 用户只能看到自己删除的文件
  * 团队管理员查看成员回收站时，校验"成员ID"与"团队ID"的关联
  * 系统管理员操作全局回收站时，记录详细审计日志
- 批量操作的二次确认：
  * 执行"全部清空"、"按条件清理"等高危操作时，弹出二次确认框
  * 显示"将彻底删除x个文件，占用yGB空间，此操作不可恢复"
  * 要求输入验证码或密码（针对重要文件）
- 支持批量操作：全部恢复、全部清空、按条件清理

**回收站中文件预览的适配：**
- 预览前的状态校验：
  * 检查文件是否已被"自动清理但数据库记录未更新"
  * 对"删除异常"的文件，允许预览但提示"文件状态异常"
- 预览资源的轻量化策略：
  * 图片预览优先使用已生成的小尺寸缩略图，不加载高清版本
  * 文档预览仅加载前5页，底部显示"回收站文件预览限制"
  * 视频预览仅显示封面图和基础信息，不支持播放
**图片预览：**
- 支持格式：jpg, png, gif, webp, svg
- 生成多种尺寸缩略图（150x150, 300x300, 800x600）
- 使用阿里云OSS图片处理服务自动生成缩略图
- 前端根据显示需求选择合适尺寸的缩略图

**文档预览：**
- PDF文件：使用PDF.js在浏览器中直接预览
- Office文档：集成Office Online或使用LibreOffice转换为PDF
- 文本文件：直接在浏览器中显示，支持语法高亮
- 支持格式：pdf, doc, docx, xls, xlsx, ppt, pptx, txt, md

**视频预览：**
- 支持格式：mp4, avi, mov, wmv, flv
- 使用FFmpeg生成视频封面图
- 大文件支持流式播放
- 可选择是否转码为web友好格式（H.264）

## 3. 文件分享系统

### 3.1 分享链接生成
**分享码与链接设计增强：**
- 生成唯一的分享码（8位随机字符串）：
  * 防碰撞校验：生成后检查file_shares表是否已存在该分享码
  * 隐性标识：前2位可融入用户ID哈希，便于问题排查
  * 语义化URL：采用简洁格式（如https://domain.com/s/abc123ef）
  * 支持生成二维码，方便线下场景快速分享

**分享参数的精细化配置：**
- 密码策略增强：
  * 支持4位数字密码或自定义密码（字母、数字、符号，长度≥6位）
  * 密码存储采用bcrypt哈希（加盐），避免明文或弱哈希
  * 可选"无密码但需登录"：仅允许系统注册用户访问
- 过期时间灵活性：
  * 固定选项：1小时、1天、7天、永久
  * 自定义日期时间（如"2025-12-31 23:59"）
  * 永久有效需二次确认，并记录操作日志
- 下载次数限制细化：
  * 区分"预览次数"和"下载次数"
  * 支持"单IP限制"（同一IP最多下载N次）
  * 分享者可实时查看剩余次数，支持中途调整限制

**分享信息存储与状态管理：**
- file_shares表字段设计：
  * id：主键
  * share_code：分享码（8位字符串）
  * file_id/folder_id：关联文件/文件夹ID
  * user_id：分享者ID
  * password_hash：访问密码哈希（nullable）
  * expire_time：过期时间（datetime，永久为null）
  * download_limit：最大下载次数（-1为无限制）
  * download_count：当前下载次数
  * permission_type：权限类型（view/download/edit/comment）
  * is_active：是否有效（是否被取消分享）
  * created_at：创建时间
  * updated_at：最后修改时间
- 批量分享支持：
  * 允许分享"多文件"或"文件夹"
  * 批量分享生成一个分享码，权限自动应用于所有子文件

### 3.2 权限控制与访问验证
**权限类型细化与场景适配：**
- 扩展权限类型：
  * 仅查看：只能预览文件，不能下载
  * 禁止预览仅下载：适用于敏感文件，避免内容被轻易查看
  * 可下载：可以预览和下载文件
  * 可评论：允许访问者添加评论（不修改文件）
  * 可编辑并保存：修改原文件
  * 可编辑并另存为：不影响原文件
- 权限与文件类型联动：
  * 非文档类型默认不支持"可编辑"权限
  * 文件夹分享时，可针对单个子文件设置例外权限

**访问验证与防滥用机制：**
- 多层验证逻辑：
  * 验证分享码是否存在且is_active=true
  * 验证是否过期（expire_time > 当前时间或永久）
  * 验证是否超过下载次数（download_count < download_limit或无限制）
  * 若有密码，验证密码正确性
  * 可选验证访问者IP是否在白名单内
- 防暴力破解与刷量：
  * 密码错误限制：连续3次错误，临时封禁该IP 10分钟
  * 高频访问限制：1分钟内同一IP访问超过20次，触发验证码验证
  * 下载速度限制：单IP每秒最大下载流量限制（如5MB/s）

**访问日志与操作追踪：**
- 日志字段精细化：
  * id：主键
  * share_id：关联file_shares表的ID
  * visitor_ip：访问者IP（可脱敏存储）
  * visitor_device：设备信息（User-Agent解析）
  * operation：操作类型（view/download/edit/comment）
  * status：操作结果（success/fail）
  * fail_reason：失败原因（如"密码错误"、"已过期"）
  * operate_time：操作时间
- 日志分析与告警：
  * 异常行为告警：如"1小时内10个不同IP下载同一分享文件"
  * 分享者可查看"访问统计"：包括总访问次数、下载次数、访问地区分布、设备分布

### 3.3 分享管理与用户体验
**分享者的动态管理能力：**
- 实时调整与取消：
  * 可随时修改分享参数：更新密码、延长过期时间、调整下载次数限制、变更权限类型
  * 支持"一键取消分享"：将is_active设为false，所有访问立即失效
- 分享列表与搜索：
  * 个人中心显示"我的分享"列表，包含文件名称、分享码、过期时间、当前状态
  * 支持按"文件名称"、"分享时间"、"状态"搜索，快速定位历史分享

**访问者的体验优化：**
- 清晰的状态提示：
  * 链接过期：显示"该分享已过期，可联系分享者重新生成链接"
  * 次数用尽：显示"下载次数已达上限，剩余0次"
  * 权限不足：提示"当前权限仅支持预览，无法下载"
- 操作引导：
  * 首次访问带密码的链接：显示"该分享受密码保护，请输入访问密码"
  * 批量下载文件夹：提供"打包下载"按钮，自动生成ZIP包

## 4. 用户权限管理

### 4.1 角色权限系统的精细化设计
**用户角色分层扩展：**
- 基础角色：
  * 普通用户：基础文件操作权限
  * 管理员：系统管理权限，可管理所有用户和文件
- 扩展角色（可由管理员决定是否开启）：
  * VIP用户：基于付费等级，拥有更高存储配额、更高单文件上传限制、优先同步/下载通道
  * 审核员：仅拥有内容审核权限（查看/标记违规文件），无文件操作权限
  * 访客用户：临时注册用户，限制存储配额、禁止创建文件夹、仅允许上传到临时空间

**角色权限矩阵：**
- 通过"角色-权限"多对多关系表（role_permissions）管理，避免硬编码
- 权限分配示例：

| 权限类别 | 普通用户 | VIP用户 | 审核员 | 系统管理员 |
|---------|---------|---------|--------|-----------|
| 上传文件 | √ | √ | × | √ |
| 创建文件夹 | √ | √ | × | √ |
| 删除自有文件 | √ | √ | × | √ |
| 查看他人分享 | √ | √ | × | √ |
| 管理所有用户 | × | × | × | √ |
| 审核违规文件 | × | × | √ | √ |

**权限控制的维度扩展：**
- 文件操作权限的细粒度拆分：
  * 基础权限：read（查看）、write（修改内容）、delete（删除）、share（分享）、rename（重命名）、move（移动）
  * 组合权限：如"编辑者"= read + write + rename，"管理员"= 所有权限
  * 特殊权限：manage_permission（管理他人权限）、restore（恢复他人删除的文件）
- 存储空间与上传限制的动态调整：
  * 配额分层：支持"个人配额 + 团队共享配额"
  * 动态调整：管理员可手动调整用户配额（如"临时扩容到20GB，有效期7天"）
  * 上传限制细化：
    - 按时间维度：每日/每月上传流量限制
    - 按内容维度：禁止上传加密压缩包、限制可执行文件
    - 按场景维度：团队内上传无类型限制，个人空间限制大文件

### 4.2 团队权限管理的深度优化
**团队角色与权限的精准映射：**
- 团队角色细化：
  * 团队所有者：完全管理权限
  * 团队管理员：成员管理、文件管理权限
  * 编辑者：拥有read + write + share权限，但无delete和成员管理权限
  * 普通成员：基础文件操作权限
  * 只读成员：仅read权限（可查看、预览，不可修改/分享）
  * 临时成员：设置有效期（如30天），到期自动降为"只读"或移除团队
- 角色权限的继承与覆盖：
  * 默认继承：子文件夹默认继承父文件夹的权限
  * 权限覆盖：支持"打破继承"——为子文件夹单独设置权限
  * 冲突解决：当成员同时属于多个团队角色，采用"权限并集"原则

**团队文件的权限边界控制：**
- 外部分享限制：
  * 严格模式：禁止任何团队文件分享到外部
  * 审核模式：分享到外部需管理员审批
  * 宽松模式：允许分享，但记录所有外部分享链接
- 成员离开团队的权限回收：
  * 立即回收：成员被移除后，立即失去所有团队文件的权限
  * 过渡期保留：可选"3天过渡期"（仅read权限）
  * 数据归属：成员离开时，其创建的文件可选择"移交他人"或"保留为团队所有"

**权限管理的可视化与操作性：**
- 权限可视化界面：
  * 团队文件夹页面显示"权限矩阵图"
  * 鼠标悬停成员名称时，显示其"有效权限来源"
- 批量权限操作：
  * 支持"按角色批量修改权限"
  * 支持"复制权限配置"：将A文件夹的权限设置一键复制到B文件夹

### 4.3 权限系统的安全性与可维护性
**权限校验与审计：**
- 多层校验机制：
  * 前端：操作前预校验（如无删除权限则隐藏删除按钮）
  * 后端：接口层强制校验（通过checkPermission函数验证）
  * 数据层：数据库查询时自动附加权限条件
- 权限变更审计日志：
  * 记录所有权限相关操作：操作人、操作类型、角色/用户、文件ID、变更前后权限、时间
  * 支持按"文件"、"用户"、"时间"检索日志

**权限缓存与性能优化：**
- 权限缓存策略：
  * 热点权限缓存：高频访问的权限缓存到Redis（过期时间10分钟）
  * 批量缓存：团队权限变更时，批量更新相关用户的权限缓存
- 权限计算优化：
  * 对深层级文件夹，通过"权限继承链缓存"避免递归查询
  * 对超大团队，权限校验时先检查用户是否为"团队所有者/管理员"（短路逻辑）

## 5. 搜索功能

### 5.1 文件搜索
**核心搜索能力：**
- 索引机制：
  * 基于MySQL全文索引构建多字段检索体系，覆盖文件名、文件描述、标签等核心属性
  * 支持中文分词处理，提升中文关键词匹配精度
  * 对高频搜索字段建立复合索引，提高查询效率
- 匹配模式：
  * 模糊搜索：支持关键词部分匹配（如输入"报"可匹配"报表.xlsx"、"日报.pdf"）
  * 支持通配符（*、?）使用，增强搜索灵活性
  * 精确匹配：通过双引号包裹关键词（如"年度报表"），仅返回完全匹配结果
- 筛选维度：
  * 文件类型：可筛选文档（doc/ppt/pdf等）、图片（jpg/png等）、视频（mp4/avi等）、压缩包（zip/rar等）等
  * 文件大小：支持自定义区间（如10MB以下、10-100MB、100MB以上）
  * 修改时间：提供快捷选项（今天、昨天、近7天、近30天）及自定义日期范围
- 结果排序：
  * 默认按"相关度+修改时间"混合排序（相关度优先，同相关度下按最近修改排序）
  * 支持手动切换为"纯修改时间"（最新在前）或"文件大小"（从大到小/从小到大）排序

**搜索体验优化：**
- 缓存策略：
  * 将TOP 100热门搜索词及结果缓存至Redis，缓存有效期2小时
  * 用户个人高频搜索词（近30天出现≥5次）本地缓存，提升重复搜索响应速度
  * 搜索结果增量缓存，减少数据库压力
- 分页机制：
  * 默认每页展示20条结果，滚动加载下一页（或分页器切换）
  * 支持"加载更多"和"跳至指定页"功能
  * 大量结果时（>1000条）提供精确计数/估算计数切换，提高性能
- 历史记录：
  * 自动保存用户近3个月搜索记录（最多100条）
  * 支持单条删除、清空历史
  * 历史记录中点击关键词可快速重新搜索
  * 智能推荐：基于历史搜索和文件访问频率推荐可能需要的文件

### 5.2 高级搜索
**多条件组合搜索：**
- 基础条件：
  * 文件名关键词：支持多关键词组合（空格分隔，默认"与"逻辑）
  * 文件类型：细分二级类型（如文档→Word/Excel/PDF，图片→PNG/JPG/PSD）
  * 支持"与/或/非"逻辑运算符组合条件（如"报表 AND 2023 NOT 草稿"）
- 进阶条件：
  * 文件大小范围：精确到字节的自定义区间设置（如5MB-20MB）
  * 时间范围：可分别指定创建时间和修改时间的起止点
  * 所有者筛选：团队场景下可指定文件所有者（从团队成员列表选择）
  * 权限范围：筛选"我拥有的"、"与我共享的"、"团队公共的"文件
  * 内容搜索：对支持的文件类型（如PDF、Office文档）进行全文内容搜索

**搜索增强功能：**
- 搜索模板：
  * 支持保存常用搜索条件为"搜索模板"（如"本周部门报表"）
  * 一键复用模板，快速执行复杂搜索
  * 支持模板分享给团队成员，提高协作效率
- 智能建议：
  * 输入时实时联想热门关键词、历史搜索词及可能的文件类型
  * 基于用户行为和文件内容提供智能搜索建议
  * 拼写纠错：自动检测并纠正可能的拼写错误
- 无结果优化：
  * 无结果时智能提示：推荐相似关键词、扩大筛选范围的建议
  * 展示相关分类文件列表，避免空白搜索结果页
  * 提供"创建新文件"快捷入口，满足找不到即创建的场景

## 6. 团队协作功能

### 6.1 团队管理
**团队创建与配置：**
- 创建流程：
  * 普通用户可创建团队（单用户最多创建5个团队），创建者自动成为团队所有者
  * 需填写必填信息：团队名称（唯一）、行业类型、团队描述（≤500字）
  * 可选配置：团队头像（支持自定义上传，默认生成随机图标）、团队公开性（私有/仅邀请可见）
- 资源配置：
  * 存储空间：默认初始配额100GB，团队所有者可申请扩容（需管理员审批）
  * 成员上限：默认最多50人，可按团队等级提升（如高级团队支持200人）
  * 功能开关：可开启/关闭团队共享空间、外部协作、版本控制等功能模块

**成员管理机制：**
- 成员邀请：
  * 邀请方式：通过邮箱发送邀请链接（有效期72小时），或生成团队邀请码（可设置使用次数/有效期）
  * 加入审批：私有团队需管理员审批，公开团队可直接加入（需团队所有者开启权限）
- 角色与权限：
  * 支持按成员批量分配角色（团队管理员/普通成员）
  * 自定义权限：团队所有者可创建子角色（如"编辑者"仅拥有写入权限，无删除权），细化权限颗粒度
- 行为追踪：
  * 成员活动日志记录：包含加入时间、最后活跃时间、文件操作记录（上传/删除/修改）、权限变更记录
  * 日志留存：保存180天，支持按成员、操作类型、时间范围筛选查询
  * 异常行为监控：检测大量删除、批量下载等敏感操作，触发管理员通知

### 6.2 协作文件管理
**共享文件夹体系：**
- 文件夹结构：
  * 团队根目录下支持多级文件夹创建（最多10级），默认包含"公共文件"（全员可读）、"部门文件"（按部门隔离）
  * 权限继承：子文件夹默认继承父文件夹权限，支持单独设置"打破继承"
  * 模板文件夹：管理员可创建文件夹模板，快速复制标准化的目录结构
- 权限控制：
  * 文件夹级权限：支持"无访问权"、"只读"（查看/下载）、"可编辑"（上传/修改）、"管理"（删除/权限分配）四级权限
  * 权限生效范围：对文件夹内所有文件及未来新增文件自动生效
  * 临时权限：支持设置有时效性的访问权限（如"3天编辑权限"），到期自动降级
- 操作审计：
  * 详细记录文件夹内所有操作：谁（用户）、何时（时间精确到秒）、操作类型（上传/移动/删除等）、操作结果（成功/失败）
  * 关键操作（如删除文件夹、批量转移文件）触发团队管理员通知
  * 支持操作回溯：查看特定文件在特定时间段内的所有操作历史

**版本控制体系：**
- 版本生成：
  * 自动保存：文件修改后每30分钟自动生成新版本；手动点击"保存版本"时即时生成
  * 版本标识：每个版本标记修改人、修改时间、修改说明（支持手动填写）
  * 版本分类：支持将重要版本标记为"里程碑版本"，防止被自动清理
- 版本管理：
  * 版本对比：支持任意两个版本的内容差异对比（文本文件显示文字差异，图片显示像素级对比）
  * 版本回滚：可一键将文件恢复至指定历史版本，回滚操作生成新的版本记录
  * 版本分支：支持从历史版本创建分支，实现并行编辑（适用于文档协作）
- 存储优化：
  * 增量存储：仅保存与上一版本的差异内容（如文档修改部分文字，仅存储修改片段），减少空间占用
  * 自动清理：超过30天且未被访问的历史版本（保留最新5个版本）自动压缩归档
  * 可手动设置重要版本为"永久保留"，避免关键历史版本被清理
  * 版本合并：支持将多个连续的小修改版本合并为一个版本，减少版本数量

## 7. 群聊功能

### 7.1 实时聊天
**核心技术实现：**
- 实时通信层：
  * 采用WebSocket协议（基于Socket.IO库）建立持久连接
  * 支持断线自动重连（重连间隔1s→3s→5s递增，最多重试10次）
  * 单连接支持每秒100+消息并发，降低服务器资源占用
  * 消息压缩传输，减少带宽消耗
- 消息存储设计：
  * MongoDB集合存储消息数据，分片存储提高查询效率
  * 每条消息包含字段：消息ID（UUID）、群聊ID、发送者ID、消息类型（文本/图片/文件）
  * 内容/资源URL、发送时间（毫秒级时间戳）、状态标识（已发送/已送达/已读）
  * 附件元数据（如图片尺寸、文件大小）、引用消息ID（回复功能）
- 消息状态流转：
  * 已发送：消息成功提交至服务器
  * 已送达：服务器确认所有在线成员接收成功
  * 已读：接收方客户端触发"已读"确认（窗口聚焦时自动确认，或手动点击确认）
  * 状态同步：通过WebSocket实时同步消息状态，支持批量状态更新

**消息类型与交互：**
- 支持消息格式：
  * 文本消息：支持基础富文本（加粗、斜体、链接）、表情符号（Unicode+自定义表情库）、代码块（语法高亮）
  * 图片消息：支持JPG/PNG/WEBP格式，自动压缩（原图≤5MB，压缩后≤1MB），预览图生成（300px缩略图）
  * 文件分享：支持关联系统内文件（直接引用存储的文件ID）和外部文件（上传至OSS后分享链接，有效期7天）
  * 显示文件名称、大小、类型图标，支持预览和直接下载
- @成员功能：
  * 输入"@"触发成员列表联想（按拼音/首字母检索）
  * 支持@全体成员（仅管理员/所有者可用）
  * 被@成员收到特殊提醒（消息标红、客户端弹窗、未读数字单独计数）
  * 点击消息中@的成员名称，可快速跳转至该成员资料页

**群聊管理机制：**
- 群聊自动关联：
  * 创建团队时自动生成同名群聊
  * 团队成员加入后自动同步至群聊（退出团队时自动移出群聊）
  * 支持团队与群聊解绑（解绑后群聊独立存在）
  * 群聊角色与团队角色自动映射（所有者、管理员、成员）
- 历史记录管理：
  * 消息永久存储（除非主动删除），新成员加入可查看入群前的完整历史
  * 支持按关键词、发送者、时间范围（今天/本周/自定义）搜索历史消息
  * 搜索结果高亮匹配内容，支持上下文跳转
  * 支持单条消息删除（仅删除自己可见的副本）或管理员批量删除（全体成员不可见）
- 文件快捷分享：
  * 集成系统文件选择器，可直接从个人/团队文件中选择文件分享至群聊
  * 自动携带文件预览和"快速保存到我的文件"入口
  * 支持拖拽文件到聊天窗口直接上传分享
  * 群聊文件自动归类，支持按类型筛选查看

### 7.2 语音聊天功能
**语音消息基础功能：**
- 录制机制：
  * 前端通过Web Audio API捕获麦克风输入，实时生成PCM音频流
  * 录制控制：长按录制按钮开始，松开自动结束；支持中途取消（滑动手指离开按钮区域）
  * 实时反馈：录制时显示动态波形图（反映音量大小）、倒计时（从60秒递减）
  * 音量过高时触发警告提示，避免失真
- 格式处理：
  * 原始录音默认保存为WebM格式（体积小、兼容性好）
  * 后台自动转码为MP3（通用播放）和AAC（高质量备份）
  * 压缩策略：采样率44.1kHz，比特率64kbps（语音清晰且体积小，1分钟约480KB）
  * 支持录音前降噪处理，提高语音清晰度

**语音消息增强处理：**
- 存储与管理：
  * 语音文件上传至OSS后生成永久访问URL，关联消息ID存储
  * 元数据记录：时长（精确到0.1秒）、文件大小、转码状态、上传时间
  * 过期策略：默认保存30天，可在团队设置中调整（7/30/90天）
  * 过期后自动清理（保留文本转写内容）
- 播放体验优化：
  * 支持倍速播放（0.5x/1x/1.5x/2x），变速不变调
  * 播放控制：点击播放/暂停、进度条拖拽、扬声器/听筒切换
  * 语音转文字：集成阿里云语音识别（支持中英双语，准确率≥95%）
  * 转写结果实时显示，可手动编辑修正
  * 未播放语音消息显示醒目标识，便于快速识别

**实时语音通话核心能力：**
- 技术架构：
  * 基于WebRTC实现点对点（P2P）通信
  * 群组通话采用SFU（选择性转发单元）架构（减少客户端带宽压力）
  * 支持最大10人同时在线通话
  * TURN/STUN服务器确保NAT穿透，提高连接成功率
- 通话功能：
  * 一对一通话：直接发起单人呼叫，支持视频通话切换（需额外开启摄像头权限）
  * 群组会议：从群聊入口发起，自动邀请群内在线成员
  * 支持手动添加离线成员（通过推送通知）
  * 通话控制：静音/取消静音、关闭/开启麦克风、切换音频设备（扬声器/耳机/蓝牙）
- 质量保障：
  * 网络自适应：根据带宽自动调整码率（8kbps-64kbps）
  * 弱网时优先保障语音连续性（牺牲部分音质）
  * 通话录音：需发起者开启并获得所有参与者同意，录音文件保存为MP3
  * 历史记录：记录通话时间、时长、参与人、录音状态，保存90天供查询

**信令系统与音频优化：**
- 信令传输：
  * 基于WebSocket的信令服务器，处理呼叫请求、接听/拒绝响应、ICE候选者交换等信令消息
  * 支持呼叫状态实时同步：呼叫中（响铃）、正在连接、通话中、已挂断、忙线（对方正在通话）、无人接听
  * 信令消息加密传输，确保通话安全
  * 支持通话转接和多方通话合并
- 音频处理技术：
  * 回声消除：通过声学回声消除（AEC）算法消除扬声器声音回传至麦克风的干扰
  * 噪音抑制：过滤背景噪音（如键盘声、空调声），保留人声清晰度
  * 编解码优化：采用Opus编码（低延迟、高压缩比），确保8kbps低码率下仍可听懂内容
  * 抖动缓冲：动态调整音频缓冲区大小（200-500ms），抵消网络抖动导致的卡顿
  * 音频设备自动检测和切换，支持蓝牙耳机、有线耳机、扬声器无缝切换

### 7.3 消息推送
**推送策略与渠道：**
- 在线状态推送：
  * 用户保持连接时，通过WebSocket实时推送消息
  * 触发客户端通知（桌面端弹窗、移动端状态栏提醒）
  * 显示消息预览（文本前20字、图片缩略图、语音时长）
  * 支持快速回复（无需打开应用即可回复简短消息）
- 离线状态处理：
  * 邮件通知：离线超过5分钟未读的消息，发送汇总邮件（包含消息列表、群聊入口链接）
  * 每日最多发送3封，避免邮件轰炸
  * 短信提醒：针对@全体成员、重要公告、语音通话请求等优先级消息
  * 触发短信通知（需用户开启该功能，每月免费额度100条）

**特殊消息推送处理：**
- 语音消息推送：
  * 通知内容显示"[语音消息] 时长XX秒"
  * 支持在通知栏直接播放（部分移动端系统支持）
  * 语音消息优先级高于普通文本消息
- 语音通话推送：
  * 来电时触发高优先级推送，显示来电者头像、名称
  * 提供"接听/挂断"按钮，支持直接操作
  * 推送持续响铃30秒（可手动关闭）
  * 未接听时转为未接来电记录并发送短信提醒
- 推送设置：
  * 支持用户自定义推送规则（如夜间免打扰时段、特定群聊静音、仅接收@我的消息）
  * 规则同步至服务器生效，确保多设备一致性
  * 支持按重要性分级推送（普通消息、@消息、通话请求）
  * 提供推送历史查看，便于追踪重要通知

**管理员配置中心：**

  * 文件类型控制：可上传文件类型白名单、危险类型黑名单（.exe、.php等）

  * 分片上传参数：默认分片大小（2MB-10MB）、并发上传数量限制（3-6个）

- 用户与权限配置：
  * 注册验证选项：邮箱验证开关、验证码有效期（5-30分钟）、注册IP限制（每IP每日最多3次）
  * 密码安全策略：最小密码长度（8-16位）、密码复杂度要求、密码过期策略（90天提醒修改）
  * 登录保护：最大失败尝试次数（5次）、账户锁定时长（15分钟）、异地登录验证
  * 团队管理：默认成员上限（50人）、团队数量限制（每用户5个）、团队空间回收策略

- 功能模块开关：
  * 预览功能：图片预览质量（标准/高清）、文档预览页数限制（全部/前20页）、视频预览清晰度（360p/720p/1080p）
  * 分享功能：默认过期时间（7天）、最长有效期（永久/1年）、外部分享审核开关
  * 同步功能：实时同步频率（1-30秒）、选择性同步开关、带宽限制（工作时间/非工作时间）
  * 语音功能：语音消息最大时长（60-300秒）、语音转文字开关、语音通话人数上限（10人）

- 系统维护配置：
  * 回收站策略：文件保留天数（7-90天）、自动清理开关、存储空间紧张时优先清理策略
  * 版本控制：历史版本保留数量（5-30个）、版本存储时长（30-180天）、增量存储开关
  * 性能优化：CDN缓存时间（公开文件/私有文件）、搜索结果缓存时长（2-24小时）
  * 系统通知：公告消息推送渠道（站内/邮件/短信）、系统维护提前通知时间（24-72小时）
  * 监控告警：存储空间告警阈值（80%/90%/95%）、系统负载告警、异常行为检测灵敏度

**配置存储方式：**
- 敏感配置：环境变量 + Kubernetes Secrets
- 业务配置：MySQL system_configs表存储
- 配置热更新：Redis发布订阅通知各服务节点
- 配置版本控制：记录配置变更历史和操作人员

### 8.3 系统监控
**监控指标：**
- 系统资源使用情况
- 文件上传下载统计
- 用户活跃度统计
- 存储空间使用统计
- 错误日志监控

## 9. 安全机制

### 9.1 数据安全
**加密存储：**
- 用户密码bcrypt加密
- 敏感配置信息加密存储
- 文件传输HTTPS加密

**访问控制：**
- JWT Token签名密钥通过`JWT_SECRET`环境变量配置
- API接口权限验证，支持RBAC角色权限控制
- 文件访问权限检查，基于用户ID和文件所有权
- 防止目录遍历攻击，严格验证文件路径
- SQL注入防护，使用参数化查询和ORM框架
- CORS跨域请求控制（允许域名通过`CORS_ORIGINS`环境变量配置）

### 9.2 系统安全

  * 搜索结果缓存：热门搜索词结果集（过期时间2小时）、个人高频搜索（用户专属缓存）
  * 文件上传状态：分片上传进度（Set类型存储）、临时令牌验证、断点续传信息
  * 分布式锁：文件合并锁、配置更新锁、关键资源访问锁（防止并发问题）
  * 计数器：访问频率限制、下载次数统计、失败尝试次数（防暴力破解）
- 浏览器缓存层：
  * IndexedDB：大文件上传状态、文件夹结构信息、下载队列状态（支持断点续传）

- CDN缓存层：
  * 静态资源：前端JS/CSS文件、图标、字体（长期缓存，版本号更新）
  * 公共文件：公开分享的文件、团队公共文件（中期缓存，带签名验证）

**MySQL优化策略：**
- 索引设计与查询优化：

  * 覆盖索引：减少回表操作，直接从索引获取数据
  * 查询重写：避免`SELECT *`，只查询必要字段
  * 执行计划分析：定期检查慢查询，优化SQL语句
- 分库分表与读写分离：
  * 水平分表：按用户ID哈希分表，解决单表数据量过大问题
  * 读写分离：主库写入，从库读取（使用ProxySQL实现透明路由）
  * 延迟从库：专用于报表查询，减轻主从库压力
- 事务与锁优化：
  * 事务粒度控制：减小事务范围，避免长事务
  * 行级锁优化：使用`SELECT ... FOR UPDATE`替代表锁
  * 死锁监控：自动检测并记录死锁情况，优化事务设计
- 定期维护：
  * 表碎片整理：定期`OPTIMIZE TABLE`，提高空间利用率
  * 统计信息更新：定期更新索引统计，优化查询计划
  * 冷热数据分离：历史数据归档，提高活跃数据查询效率

**MongoDB优化策略：**
- 集合设计：
  * 消息集合分片：按群聊ID分片，提高并发写入性能
  * TTL索引：自动清理过期数据（如临时消息、日志记录）
  * 复合索引：针对消息搜索场景优化（发送者+时间+内容）
- 查询优化：
  * 投影查询：只返回必要字段，减少网络传输
  * 聚合管道优化：尽早过滤，减少处理数据量
  * 游标批处理：大结果集分批处理，避免内存溢出
- 连接池管理：
  * 动态连接池：根据负载自动调整连接数
  * 超时设置：避免连接长时间占用
  * 错误重试：自动重连机制，提高可用性

**Redis优化策略：**
- 内存管理：
  * 数据结构选择：根据场景选择最优数据结构（如分片上传进度用Set）
  * 过期策略：设置合理TTL，避免内存泄漏
  * maxmemory策略：配置LRU/LFU淘汰策略，保护关键数据
- 持久化配置：
  * AOF+RDB混合持久化：兼顾数据安全和恢复速度
  * 定时BGSAVE：避免主进程阻塞
  * 主从复制：数据冗余备份，提高可用性
- 集群与分片：
  * Redis Cluster：按key哈希自动分片
  * 读写分离：主节点写入，从节点读取
  * 哨兵模式：自动故障转移，提高可用性

### 10.3 文件存储优化
**OSS存储策略：**
- 分层存储架构：
  * 热数据层：频繁访问文件，使用高性能存储，CDN加速
  * 温数据层：近期文件，标准存储，按需CDN加速
  * 冷数据层：长期未访问文件，低频访问存储，降低成本
  * 归档层：历史版本、已删除文件备份，归档存储
